# optigo

Проект по изучению инструментов профилирования Go (`pprof`, `trace`) и итеративной оптимизации HTTP-сервиса.

## Стек технологий
* **Language:** Go 1.25+
* **Framework:** [Echo v4](https://github.com/labstack/echo)
* **Profiling:** net/http/pprof
* **Load Testing:** [hey](https://github.com/rakyll/hey)

---

## Проект: Нагрузочное тестирование и оптимизация

### Цель
Создать масштабируемый сервис, выявить узкие места (CPU/RAM) с помощью профилировщика и оптимизировать их, используя бенчмарки для подтверждения результата.

---

## Этап 1: Baseline

### Особенности реализации
* **Динамическая компиляция регулярных выражений:** `regexp.MustCompile` вызывается внутри хендлера на каждый входящий запрос.
* **Лишние аллокации:** Использование `fmt.Sprintf` в циклах для конкатенации строк.
* **Неоптимальная работа с памятью:** Создание новых слайсов без предварительного выделения емкости (`make`).

> **Вывод:** Сервис показывает базовую производительность. Высокие задержки обусловлены неэффективным использованием CPU на перекомпиляцию регулярок и частыми аллокациями памяти.

## Этап 2: Global Regexp & Memory Pre-allocation

### Изменения:
- Вынос `regexp.MustCompile` в глобальную переменную (компиляция один раз при старте).
- Предварительное выделение памяти под слайс: `make([]string, 0, len(req.Names))`.
- Замена `fmt.Sprintf` на конкатенацию строк.

> **Вывод:** Производительность выросла, но профилирование `top20` показало, что `regexp` всё еще занимает ~6% CPU из-за внутренних блокировок.

## Этап 3: No Regexp (Byte Comparison)

### Изменения:
- Полный отказ от пакета `regexp` в пользу прямой проверки первого байта строки (`name[0] >= 'A'`).
- Это позволило устранить конкуренцию за мьютекс внутри движка регулярных выражений.

> **Вывод:** Для простых проверок в высоконагруженных системах следует избегать регулярных выражений. Прямая работа с байтами и строками дает максимальный прирост RPS и снижает задержки.

## Этап 4: Memory Optimization (sync.Pool)

### Изменения:
- Внедрен `sync.Pool` для переиспользования слайсов `[]string`.
- Цель: снизить количество аллокаций и нагрузку на Garbage Collector (GC).

> **Вывод:** После внедрения пула основная нагрузка переместилась на создание динамических строк при конкатенации и парсинг JSON, в то время как аллокации слайсов были минимизированы.

---

### Команда запуска теста
```bash
hey -n 20000 -c 50 -m POST -T "application/json" -D data.json http://localhost:8082/process
```

### Сводные результаты тестов

| Этап       | Метод | RPS (средний) | P99 Latency | Прирост (RPS) |
|:-----------| :--- | :--- | :--- | :--- |
| **Этап 1** | Baseline (Naive) | 11,521 | 0.0216s | - |
| **Этап 2** | Global Regexp + Make | 15,100 | 0.0171s | +31% |
| **Этап 3** | **No Regexp (Byte check)** | **17,505** | **0.0126s** | **+52%** |
| **Этап 4** | No Regexp + sync.Pool | 16,834 | 0.0155s | Оптимизация RAM |

> **Примечание к этапу 4:** Внедрение `sync.Pool` позволило стабилизировать работу с памятью (снизить количество аллокаций), сохранив высокую производительность, сопоставимую с лучшим результатом Этапа 3.

---

### Итоговые выводы
1. **Библиотеки не всегда бесплатны:** Даже стандартный `regexp` может стать узким местом из-за внутренних блокировок.
2. **Профилирование это база:** Без `pprof` невозможно определить, куда уходят ресурсы (CPU vs RAM).
3. **Аллокации решают всё:** Использование `sync.Pool` и `make` с заранее известной емкостью значительно облегчает жизнь Garbage Collector в высоконагруженных системах.

### 🛠 Как запустить локально

1. **Генерация тестовых данных**:
   ```bash
   go run gen.go
   ```
2. **Запуск сервера**:
   ```bash
   go run main.go api.go
   ```
3. **Сбор профиля (CPU) во время нагрузки**:
   ```bash
   go tool pprof http://localhost:8082/debug/pprof/profile?seconds=10
   ```
